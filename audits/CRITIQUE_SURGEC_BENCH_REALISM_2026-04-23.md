# CRITIQUE_SURGEC_BENCH_REALISM_2026-04-23

Scope: `libs/tools/surgec/benches/*.rs`

Bottom line: `compile_throughput.rs` and `fuse_scaling.rs` are sub-step microbenches with toy synthetic inputs; they are mostly honest if cited only as compiler-internal measurements. `specialize_vs_generic.rs` and `vs_competition.rs` are not safe to cite as runtime or end-to-end proof: one measures a tiny CPU interpreter hot loop, and the other feeds Criterion cached milliseconds instead of timing the operation it names.

## Bench matrix

| File | Measures what the name claims? | Dispatch + readback? | Input realism | Apples-to-apples baseline? |
| --- | --- | --- | --- | --- |
| `compile_throughput.rs` | **Partially.** `parse_*` and `compile_*` are honest; `end_to_end` is only in-memory parse+compile, not file-backed or scan end-to-end. | **No.** Compiler-only. | **Toy.** Single rule or up to 1000 repeated trivial rules. | Not applicable for compile-only benches. |
| `fuse_scaling.rs` | **Mostly yes.** It measures fuse-plan work only. | **No.** Pure compiler pass. | **Toy.** Repeated identical rules, max 1000. | Not applicable for fuse-only bench. |
| `specialize_vs_generic.rs` | **No.** It sounds like runtime specialization performance, but measures `vyre_reference::reference_eval` on the CPU reference interpreter. | **No.** No backend dispatch, no upload, no readback. | **Toy.** 1 word, 12 words, and an 8-byte base64 sample. | **No.** No CPU-vs-GPU or same-path baseline; both arms use the same interpreter. |
| `vs_competition.rs` | **No.** The expensive work runs once in `run_matrix()`, then Criterion benches replay cached durations. | `measure_surgec()` does include real scan work, but Criterion does not re-measure it. | **Mixed.** Real checked-in corpora are 1/10/100 MB; 1 GB and 100 GB placeholders are skipped. | **No.** Cells include published baselines and mismatched domains/tool shapes. |

## Findings

1. **HIGH** | `libs/tools/surgec/benches/compile_throughput.rs:70-79` | `end_to_end` measures only `surgec::parse_str(src)` plus `surgec::compile(&doc)` on an already-in-memory string. That is not end-to-end `surgec` behavior in the normal sense: no file loading, no `compile_paths`, no CLI/config overhead, no backend validation, and no scan dispatch/readback. Because the file-level banner says "Compilation throughput benchmarks" (`:1-4`), this is not fraudulent by itself, but the bare `end_to_end` name is easy to overquote. | Make it rigorous by renaming it to `parse_and_compile_in_memory`, and add a separate file-backed bench using `compile_paths` over representative real rule packs with cold/warm filesystem behavior reported explicitly.

2. **MEDIUM** | `libs/tools/surgec/benches/compile_throughput.rs:8-34,42-79` | The compile inputs are synthetic toy shapes: one trivial rule or repeated copies of the same two-signal rule, capped at 1000 copies. This exercises parser/IR construction overhead, not realistic multi-megabyte heterogeneous rulepacks. | Make it rigorous by benchmarking checked-in real rulesets, reporting source bytes / rule count / IR node count per case, and adding at least one multi-megabyte rules corpus instead of repeated template expansion.

3. **MEDIUM** | `libs/tools/surgec/benches/fuse_scaling.rs:15-29` | `fuse_scaling` times only `surgec::compile::fuse::fuse_document_programs(...)` plus `plan.optimize(64)` after the document has already been parsed and compiled (`:18-20`). The filename is honest about the sub-step, but this is a hot-loop compiler-pass microbench, not compile throughput and not runtime scan performance. It has no dispatch/readback, and the generated corpus is again a repeated identical rule template (`:3-12`). | Make it rigorous by feeding fusion with compiled documents captured from real rulesets, publishing source/IR sizes beside timings, and keeping this bench explicitly labeled as a fuse-pass microbench rather than using it in any broader speed narrative.

4. **CRITICAL** | `libs/tools/surgec/benches/specialize_vs_generic.rs:132-231` | The benchmark claims `specialize_vs_generic`, but every timed path calls `vyre_reference::reference_eval(...)` (`:149-150`, `:159-160`, `:179-180`, `:189-190`, `:217-218`, `:227-228`). `vyre_reference::reference_eval` is the pure Rust reference interpreter, not a backend dispatch path (`libs/performance/matching/vyre/vyre-reference/src/interp.rs:37-49`). So the bench omits kernel launch, buffer upload, device execution, and readback entirely while still reading like a runtime speedup comparison. The inputs are tiny: a single `u32`, twelve `u32`s, and the 8-byte string `TWFuTWFu` (`:135-141`, `:165-171`, `:196-209`). This is a textbook "hot-loop inner only" benchmark with fraudulent-number risk if cited as surgec/Vyre execution speedup. | Make it rigorous by benchmarking the same generic and specialized programs through the real backend, including host-to-device transfer, dispatch, and output readback, on multi-megabyte representative inputs. If the intended claim is "reference interpreter specialization helps," rename the file and benchmark IDs to say exactly that.

5. **CRITICAL** | `libs/tools/surgec/benches/vs_competition.rs:14-17,162-191` | Criterion is not measuring the competition workload. `benchmark_report()` populates a `OnceLock` from `run_matrix()` once, then `bench_competition()` feeds Criterion only the cached `warm_ms` scalars via `b.iter_custom(...)`. The reported benchmark timing is therefore a replay of previously measured numbers, not a fresh measurement of surgec or any competitor under Criterion. That destroys Criterion's statistical value and makes the output dangerously easy to misread as a live benchmark. | Make it rigorous by moving the real work into each timed closure, or delete Criterion from this file and keep it as a report generator only. A benchmark must measure the operation it names, not a stored float.

6. **CRITICAL** | `libs/tools/surgec/benches/vs_competition.rs:40-55,165-193` | The competition cells are not apples-to-apples, yet they are all benchmarked. `speedup` is computed before the comparability verdict, every cell is pushed regardless of comparability, and `bench_competition()` iterates over all cells without filtering. Supporting evidence in the harness shows several mismatches: Hyperscan is regex-byte throughput and even warns its measured mode needs a corpus DB (`libs/performance/matching/vyre/benches/competition/competitors/hyperscan.rs:39-78`); Nuclei is a published network-assets baseline (`.../nuclei.rs:22-41`); CodeQL and Snyk are published wall-clock baselines from unrelated external studies with no corpus anchor (`.../codeql.rs:22-43`, `.../snyk.rs:22-41`). Even the realistic corpus side is limited to checked-in 1/10/100 MB trees because download corpora are skipped (`libs/tools/surgec/benches/vs_competition.rs:34-37`; `libs/performance/matching/vyre/benches/competition/corpora/manifest.toml:3-28,30-60`). This is a direct apples-to-oranges risk. | Make it rigorous by benchmarking only measured, same-domain, same-corpus, same-ruleset cells; filtering out incomparable cells before both speedup computation and publication; and ensuring surgec and the comparator run with equivalent process/model boundaries.

## Verdict by file

- `compile_throughput.rs`: acceptable only as a compiler microbench suite, not as an end-to-end benchmark.
- `fuse_scaling.rs`: acceptable only as a fuse-pass microbench.
- `specialize_vs_generic.rs`: not acceptable as currently named or scoped; it measures the wrong execution path for the implied claim.
- `vs_competition.rs`: not acceptable as a benchmark; it is a report replay wrapped in Criterion and mixes incomparable baselines.

# AUDIT C — Organization / LAW 7 (SQLite-Unix)
**Date:** 2026-04-18  
**Scope:** vyre workspace — `libs/performance/matching/vyre/`  
**Standard:** LAW 7 — every file has ONE function, every module has ONE responsibility, no god files.  
**Mode:** read-only

---

## Executive Summary

**35 findings** (12 CRITICAL, 14 HIGH, 9 MEDIUM).  
No file is above 1 000 LOC, but 8 files breach the 500-LOC hard limit.  
The most systemic issue is section-comment proliferation in `vyre-conform` tests: 10 "corruption vectors" in a single 424-LOC file is a textbook violation of Law 7. Secondary: three God-module candidates, a mini-interpreter duplicated inside `vyre-core/src/fuzz.rs` that shadows `vyre-reference`, and two embedded WGSL shaders living as 100-LOC string literals in a bench file.

### ORG-10 — `vyre-core/src/lib.rs` — MEDIUM — 182 LOC (re-exports 9 public submodules + contains non-trivial `allow` commentary)

**Finding:** While 182 LOC is under the 500-LOC hard cap, `lib.rs` contains two logically distinct sections: (1) crate-wide `#![allow(...)]` directives with 12 annotated exemptions (~50 LOC), and (2) 9 module declarations with inline doc comments (~100 LOC). The `allow` directives are part of a separate `LINTS.md`-style policy that should either live in `Cargo.toml` (via `[lints]`) or a `clippy.toml`, not in `lib.rs` where they compete with module docs for attention.

**Fix:** Move `#![allow(...)]` block to `Cargo.toml` `[lints.clippy]` table (Rust 1.73+). `lib.rs` becomes pure module declarations.

For publishable crates on crates.io this is correct — each crate's consumers read its own changelog. **Finding:** The root `CHANGELOG.md` covers the `vyre` crate only but is positioned at workspace root, implying workspace scope. ## §5 — Crate Purpose Clarity

### ORG-23 — `vyre` (vyre-core) — "GPU compute IR *and also* a standard operation library" — HIGH

**Cargo.toml description:** `"GPU compute intermediate representation with a standard operation library"`

The "and also" is explicit. The crate publishes `pub mod ops` which is an entire L1+L2 operation library (string matching, hash, compression, buffer, stats, workgroup). `vyre-std` *also* exists and ships L2 operations. The line between what lives in `vyre::ops` vs `vyre_std` is not enforced by a module boundary — it is a convention that can drift.

**Fix:** The crate description should be `"GPU compute intermediate representation and primitive ops"`. All L2+ composition helpers must migrate to `vyre-std`. Document the tier boundary (`L1 ops → vyre`, `L2 composition → vyre-std`) in `ARCHITECTURE.md` §"Crate responsibilities."

---

### ORG-24 — `vyre-std` — "compositional helpers *and also* a DFA assembly pipeline" — HIGH

**lib.rs opening:** `"vyre-std ships the L2 layer: compositional helpers built from those primitives plus the full GPU DFA assembly pipeline (Thompson NFA → subset construction → Hopcroft → pack)."`

Two distinct systems: (1) general L2 composition helpers, and (2) a full NFA→DFA compiler pipeline (`pattern::dfa_assemble`, `pattern::regex_to_nfa`, `pattern::dfa_assemble`). The DFA pipeline is large enough to be a standalone crate (`vyre-patterns`).

**Fix:** Extract `vyre-std/src/pattern/` into `vyre-patterns` crate. `vyre-std` becomes purely compositional arithmetic/buffer helpers. The lib.rs "one-line consumer API" for DFA assembly moves to `vyre-patterns`.

---

### ORG-25 — `vyre-core/src/fuzz.rs` exported as public API — CRITICAL

`vyre-core/src/lib.rs` exports `pub mod fuzz`. The `fuzz` module contains a LCG PRNG, a mini-interpreter, program generators — all test/bench tooling. This tooling is part of the public stable API surface of the `vyre` crate on crates.io.

Publishing test helpers as stable public API violates single-responsibility (the crate is "GPU IR", not "GPU IR and a test generator"). Any downstream code can `use vyre::fuzz::gen_program` and expect semver guarantees.

**Fix:** Gate behind `#[cfg(any(test, feature = "fuzz-helpers"))]` or (better) move to a `vyre-test-support` workspace-private crate with `publish = false`.

## §7 — Docs/ Audit Reporting Organization

### ORG-32 — `docs/audits/` has no per-run naming convention — HIGH

Current contents:
```
docs/audits/MASTER_PLAN_RELEASE.md   ← planning doc, not an audit
docs/audits/MASTER_PLAN.md             ← planning doc, not an audit
docs/audits/ROADMAP_PERFORMANCE.md     ← roadmap doc, not an audit
docs/audits/SWEEP.md                   ← task tracker, not an audit
docs/audits/pre-sweep-errors.txt       ← raw error dump, no structure
```

Zero files follow a `AUDIT_<date>_<category>.md` naming pattern. The directory is misused as a dumping ground for planning artifacts. An auditor cannot determine which file corresponds to which audit run or which audit category.

**Fix:**
1. Move `MASTER_PLAN*.md` → `docs/planning/`  
2. Move `ROADMAP_PERFORMANCE.md` → `docs/roadmap/`  
3. Move `SWEEP.md` → `docs/planning/` (it's a task tracker)  
4. Move `pre-sweep-errors.txt` → `docs/audits/AUDIT_2026-04-17_pre-sweep-errors.txt` (or delete if superseded)  
5. Enforce naming: every audit file = `AUDIT_<YYYY-MM-DD>_<category>.md` where category ∈ {sec, org, perf, deps, api}.
